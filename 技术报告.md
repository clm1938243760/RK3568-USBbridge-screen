# RK3568 USB Bridge Screen 技术报告

## 1. 项目背景

本项目运行在正点原子 ATK-DLRK3568 / RK3568 嵌入式 Linux 开发板上。开发板通过 USB 与 Windows 电脑连接，在系统层面模拟多个 USB 设备，实现医疗体检报告场景中的自动化录入、打印捕获、PDF 转换和上传。

系统主要能力包括：

- 模拟 USB HID 键盘，将患者信息自动录入到 Windows 端业务软件。
- 模拟 USB HID 鼠标，按预设坐标自动点击窗口控件。
- 模拟 USB 打印机，接收 Windows 端打印出来的报告数据。
- 将捕获到的打印数据转换成 PDF。
- 将 PDF 上传到服务端接口。
- 通过扫码枪读取患者条码并调用患者信息接口。
- 在本地 480x320 小屏上显示当前流程状态、扫码状态、项目选择、HID 录入、打印和上传进度。

原系统曾经使用 Python 实现，本仓库是 C 语言加 QML 的实现版本。C 版本的目标是降低运行时依赖、提升启动稳定性，并更适合在 RK3568 Buildroot 环境中长期部署。

## 2. 设计目标

本项目不是一个通用桌面程序，而是一个面向现场使用的嵌入式自动化终端。设计时重点考虑以下目标：

- **稳定启动**：开机后自动配置 USB Gadget，并自动拉起扫码、HID、打印、上传和 UI 进程。
- **流程可视化**：操作人员可以从小屏幕看到系统当前是在等待扫码、查询接口、选择项目、录入、等待打印、转换 PDF 还是上传。
- **低耦合**：业务进程不直接控制 UI，而是统一写状态文件，UI 只负责读取状态并显示。
- **现场可调试**：所有关键进程都有独立日志，队列文件和状态文件可以直接查看。
- **适配 480x320 小屏**：最终部署目标是 480x320 显示屏。当前接 HDMI 大屏调试时，界面仍保持 480x320 面板尺寸。
- **避免原厂桌面干扰**：停止原厂 `systemui`，保留 Weston 作为 Wayland 显示服务，避免 Qt 直接走 `linuxfb` 时在 RK3568 上崩溃。

## 3. 总体架构

系统由多个独立进程组成，通过目录、队列文件和 JSON 状态文件协同工作。

完整数据流如下：

```text
扫码枪输入
  -> scan_patient_capture
  -> 调用患者信息 API
  -> 解析接口 JSON
  -> 生成项目选择状态
  -> 用户通过按键确认项目
  -> 生成 /root/usb_bridge/form_queue/form_*.json
  -> hid_executor
  -> /dev/hidg0 模拟键盘
  -> /dev/hidg1 模拟鼠标
  -> Windows 业务软件完成录入
  -> 状态进入 wait_print
  -> Windows 打印报告到 USB 打印机
  -> printer_capture 从 /dev/g_printer0 捕获打印数据
  -> report_uploader 调用 Ghostscript 生成 PDF
  -> report_uploader 调用 curl 上传 PDF
  -> UI 显示完成并返回等待扫码
```

UI 的状态来源是：

```sh
/root/usb_bridge/state/ui_status.json
```

QML 界面每 500ms 读取一次该文件，根据 `phase` 字段刷新画面。

## 4. 主要文件说明

| 文件 | 作用 |
| --- | --- |
| `scan_patient_capture.c` | 读取扫码枪输入事件，调用患者接口，解析 JSON，处理项目选择，生成 HID 表单任务。 |
| `hid_executor.c` | 消费表单任务，通过 `/dev/hidg0` 和 `/dev/hidg1` 发送键盘、鼠标 HID 报告。 |
| `printer_capture.c` | 监听打印等待队列，从 `/dev/g_printer0` 捕获 Windows 打印数据。 |
| `report_uploader.c` | 使用 `gs` 将报告转换为 PDF，并使用 `curl` 上传。 |
| `usb_bridge_common.c/.h` | 公共 JSON 解析、日志、Base64、原子写文件、UI 状态写入。 |
| `api_config.c/.h` | 读取 `/root/usb_bridge/config/api.conf` 和环境变量覆盖项。 |
| `usb_bridge_ui.qml` | QML 状态界面，显示 480x320 固定尺寸面板。 |
| `bridge_ui_qml` | QML 启动脚本，设置 Qt 平台和 Wayland 窗口装饰参数。 |
| `bridge_ui.c` | C 语言 framebuffer 备用 UI，仅作为紧急兜底。 |
| `S50usbdevice` | USB Gadget 初始化脚本。 |
| `S98usbbridge` | 主服务脚本，负责启动、停止、监督所有 worker 和 UI。 |
| `sendkey.c` | 测试工具，用于模拟 Linux input 按键事件。 |
| `install_to_board.sh` | 板端安装脚本。 |
| `api.conf.example` | API 和输入设备配置示例。 |

## 6. USB Gadget 设计

系统需要开发板对 Windows 电脑表现为一个复合 USB 设备，包含：

- HID Keyboard
- HID Mouse
- USB Printer Gadget

`S50usbdevice` 负责初始化 USB Gadget。启动成功后，应能看到：

```sh
/dev/hidg0
/dev/hidg1
/dev/g_printer0
```

其中：

- `/dev/hidg0`：键盘 HID 设备。
- `/dev/hidg1`：鼠标 HID 设备。
- `/dev/g_printer0`：USB 打印机 Gadget 设备。

检查命令：

```sh
ls -l /dev/hidg0 /dev/hidg1 /dev/g_printer0
```

如果设备不存在，可重启 USB Gadget：

```sh
/etc/init.d/S50usbdevice restart
sleep 2
ls -l /dev/hidg0 /dev/hidg1 /dev/g_printer0
```

## 7. 扫码与患者接口流程

扫码流程由 `scan_patient_capture` 负责。

主要步骤：

1. 从 Linux input 设备读取扫码枪按键事件。
2. 将按键事件还原成条码字符串。
3. 调用患者查询接口。
4. 保存接口原始响应到 `/root/usb_bridge/api_raw`。
5. 解析 JSON 中的患者记录。
6. 如果存在多个检查项目，进入项目选择状态。
7. 用户确认项目后，生成 HID 表单任务。

现场验证过的扫码设备：

```conf
SCANNER_DEV=/dev/input/event8
```

注意：`/dev/input/event*` 编号可能在重启或 USB 重新枚举后变化。如果扫码无响应，需要查看：

```sh
cat /proc/bus/input/devices
tail -f /root/usb_bridge/logs/scan_patient_capture.log
```

## 8. 项目选择流程

患者接口可能返回多条记录，每条记录可能对应不同的 `exam_item`。系统会提取唯一的 `exam_item` 列表，在 UI 上显示为可选项目。

选择按键支持：

| 动作 | 默认键值 |
| --- | --- |
| 上一个 | `KEY_LEFT=105`，也支持方向上/左 |
| 下一个 | `KEY_RIGHT=106`，也支持方向下/右 |
| 确认 | `KEY_ENTER=28`，也支持小键盘 Enter |

现场验证过的项目选择输入设备：

```conf
PROJECT_INPUT_DEV=/dev/input/event5
```

也支持外接 USB 键盘进行方向键和 Enter 选择。

测试按键工具：

```sh
/root/usb_bridge/bin/sendkey /dev/input/event5 106
/root/usb_bridge/bin/sendkey /dev/input/event5 105
/root/usb_bridge/bin/sendkey /dev/input/event5 28
```

## 9. HID 自动录入流程

`hid_executor` 消费：

```sh
/root/usb_bridge/form_queue/form_*.json
```

处理完成后移动到：

```sh
/root/usb_bridge/form_done
```

HID 自动录入包括：

- 鼠标移动和点击。
- ASCII 字符输入。
- 中文文本通过 Windows PowerShell 剪贴板粘贴。
- 为避免 Windows 中文输入法干扰，ASCII 输入前会临时处理 CapsLock 状态。

常用排查命令：

```sh
ps | grep hid_executor | grep -v grep
ls -l /root/usb_bridge/form_queue
ls -l /root/usb_bridge/form_done
tail -n 120 /root/usb_bridge/logs/hid_executor.log
```

如果 UI 显示录入完成但 Windows 没有输入，优先检查：

- `hid_executor` 是否运行。
- `/dev/hidg0` 和 `/dev/hidg1` 是否存在。
- 表单任务是否仍停留在 `form_queue`。
- Windows 端窗口焦点和点击坐标是否正确。
- Windows 输入法是否处于中文状态。

## 10. 打印捕获与报告上传

`printer_capture` 负责监听报告打印队列并读取：

```sh
/dev/g_printer0
```

捕获完成后，报告数据进入 PDF 队列。

`report_uploader` 负责：

1. 调用 Ghostscript `gs` 转换 PDF。
2. 调用 `curl` 上传 PDF。
3. 更新 UI 状态为 `converting`、`uploading`、`done` 或 `error`。

常用排查命令：

```sh
tail -n 120 /root/usb_bridge/logs/printer_capture.log
tail -n 120 /root/usb_bridge/logs/report_uploader.log
ls -l /root/usb_bridge/report_pdf_queue
ls -l /root/usb_bridge/report_uploaded
ls -l /root/usb_bridge/report_error
```

## 11. UI 状态协议

所有 worker 通过公共函数写入：

```sh
/root/usb_bridge/state/ui_status.json
```

典型内容：

```json
{
  "time": "2026-05-07 16:47:31",
  "phase": "printing",
  "scan_text": "p2604270002",
  "patient_id": "P...",
  "report_no": "R...",
  "selected_index": 0,
  "project_count": 1,
  "project_options": ["检查项目"]
}
```

常见字段说明：

| 字段 | 说明 |
| --- | --- |
| `time` | 状态更新时间。 |
| `phase` | 当前流程阶段，UI 主要根据它切换显示。 |
| `scan_text` | 当前扫码内容。 |
| `patient_id` | 患者 ID。 |
| `report_no` | 报告号。 |
| `selected_index` | 当前选中的项目索引。 |
| `project_count` | 项目数量。 |
| `project_options` | 检查项目列表。 |

已定义的 `phase`：

```text

ready_input
inputting
wait_print
printing
print_done
converting
uploading
done
error
```

## 12. UI 显示方案

板子原厂桌面相关服务：

```sh
/etc/init.d/S49weston
/etc/init.d/S50systemui
```

现场测试后确定的稳定方案：

- 保留 `S49weston`，让 Weston 作为 Wayland 合成器。
- 停止并禁用 `S50systemui`，避免原厂桌面界面残留和抢占显示。
- 自定义 QML 使用 Wayland 平台运行。

推荐配置：

```conf
BRIDGE_UI_MODE=qml
BRIDGE_UI_QML_PLATFORM=wayland
BRIDGE_UI_NO_FALLBACK=1
BRIDGE_UI_START_DELAY=8
BRIDGE_DISABLE_FACTORY_DESKTOP=1
BRIDGE_DISABLE_WESTON=0
BRIDGE_UI_MAX_FAILS=5
```

`usb_bridge_ui.qml` 的显示策略：

- 创建全屏黑色 Wayland 窗口，覆盖 Weston 桌面残留。
- 内部 UI 面板固定为 `480x320`。
- 在 HDMI 大屏调试时，黑底居中显示 480x320 面板。
- 以后接 480x320 小屏时，面板刚好铺满整个屏幕。

`bridge_ui_qml` 会在 Wayland 下设置：

```sh
QT_WAYLAND_DISABLE_WINDOWDECORATION=1
```

用于避免 Weston 为窗口加标题栏或边框。

## 13. 启动脚本设计

主启动脚本：

```sh
/etc/init.d/S98usbbridge
```

启动流程：

1. 创建运行目录。
2. 写入初始 `ui_status.json`，状态为 `service_start`。
3. 停止旧 worker 和残留 UI 进程。
4. 停止并禁用原厂 `systemui`。
5. 保留或启动 Weston。
6. 启动 QML UI。
7. 等待 `/dev/hidg0`、`/dev/hidg1`、`/dev/g_printer0`。
8. 启动 `hid_executor`。
9. 启动 `scan_patient_capture`。
10. 启动 `printer_capture`。
11. 启动 `report_uploader`。

`S98usbbridge` 使用 supervisor 模式拉起 worker。如果进程异常退出，会自动重启。QML UI 连续崩溃达到 `BRIDGE_UI_MAX_FAILS` 后，UI supervisor 会停止，避免无限刷屏，但后台业务服务仍可继续运行。

停止服务时，脚本会清理：

```sh
scan_patient_capture
hid_executor
printer_capture
report_uploader
bridge_ui_qml
bridge_ui
qmlscene.*usb_bridge_ui.qml
qml.*usb_bridge_ui.qml
```

这样可以避免多个 QML 实例叠加显示。

## 18. 常见问题与处理

### 18.1 QML 报 `Failed to create wl_display`

原因通常是 Weston 未运行，但 QML 使用了 Wayland 平台。

处理：

```sh
chmod +x /etc/init.d/S49weston
/etc/init.d/S49weston start
sed -i '/BRIDGE_UI_QML_PLATFORM/d' /root/usb_bridge/config/ui.conf
echo 'BRIDGE_UI_QML_PLATFORM=wayland' >> /root/usb_bridge/config/ui.conf
/etc/init.d/S98usbbridge restart
```

### 18.2 QML 反复 `Aborted rc=134`

可能原因：

- Qt 平台插件选择错误。
- Weston 未运行。
- 强制使用 `linuxfb` 触发 Rockchip DRM/fbdev 问题。
- QML 文件语法错误。

排查：

```sh
tail -n 200 /root/usb_bridge/logs/bridge_ui_qml.log
cat /root/usb_bridge/config/ui.conf
ps | grep -E 'weston|qml|bridge_ui' | grep -v grep
```

### 18.3 屏幕上出现两个 UI

说明有残留的 `qmlscene` 或 `qml` 实例。

处理：

```sh
/etc/init.d/S98usbbridge stop
pkill -9 -f qmlscene
pkill -9 -f qml
pkill -9 -f /root/usb_bridge/bin/bridge_ui_qml
rm -f /tmp/bridge_ui.pid
/etc/init.d/S98usbbridge restart
```

### 18.4 原厂桌面残留

保留 Weston 是为了 Wayland 显示，但必须关闭 `systemui`。

处理：

```sh
/etc/init.d/S50systemui stop 2>/dev/null
chmod -x /etc/init.d/S50systemui
killall systemui 2>/dev/null
/etc/init.d/S98usbbridge restart
```

### 18.5 扫码无响应

检查扫码枪设备号是否变化：

```sh
cat /proc/bus/input/devices
tail -f /root/usb_bridge/logs/scan_patient_capture.log
```

如有变化，修改：

```sh
/root/usb_bridge/config/ui.conf
```

中的：

```conf
SCANNER_DEV=/dev/input/eventX
```

### 18.6 HID 没有输入到 Windows

优先检查：

```sh
ps | grep hid_executor | grep -v grep
ls -l /dev/hidg0 /dev/hidg1
ls -l /root/usb_bridge/form_queue
tail -n 120 /root/usb_bridge/logs/hid_executor.log
```

如果 HID 日志显示已发送事件，但 Windows 没有输入，需要检查：

- Windows 窗口焦点。
- 鼠标点击坐标。
- Windows 输入法状态。
- USB HID 是否重新枚举。

## 19. 维护注意事项

- 本仓库通过 `.gitattributes` 固定 C、头文件、QML、Makefile 和 shell 脚本使用 LF 换行，避免 Windows CRLF 导致板端脚本执行异常。
- `AI_HANDOFF.md` 是详细交接记录，适合开发和排障参考；正式技术报告以本文档为准。
- 不建议生产环境启用旧 C framebuffer UI，除非 QML 无法启动。
- 不建议关闭 Weston，除非明确要测试 `linuxfb` 或其他显示后端。
- API 地址、上传地址、输入设备号应通过配置文件修改，不应硬编码到程序中。
- 现场部署前应确认 `/dev/hidg0`、`/dev/hidg1`、`/dev/g_printer0` 都存在。

## 20. 总结

本项目将 RK3568 开发板改造成一个面向 Windows 业务软件的 USB 自动化桥接终端。它通过 USB Gadget 同时提供 HID 键盘、HID 鼠标和 USB 打印机能力，通过 C 语言 worker 完成扫码、接口查询、自动录入、打印捕获、PDF 转换和上传，通过 QML 在本地 480x320 屏幕上显示流程状态。

当前推荐的显示方案是“Weston 保留，systemui 禁用，QML 走 Wayland”。这样既避免了原厂桌面干扰，又绕开了 RK3568 上 `linuxfb` 可能触发的崩溃问题。UI 使用全屏黑底覆盖桌面残留，同时保持 480x320 面板尺寸，为后续接入 480x320 小屏做好适配。
